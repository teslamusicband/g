# Обзор ключевых исследований по S2ST (arXiv)

Прежде чем проектировать архитектуру — вот основные линии исследований, которые определяют текущий SOTA:

**Эволюция подходов:**
- Первая прямая (direct) модель — Translatotron: она предсказывает спектрограммы напрямую из входной речи без промежуточного текста и показывает разумную точность перевода и естественность речи. Позже Translatotron 2 добавил вспомогательный фонемный декодер для повышения качества перевода.
- Ключевой сдвиг — переход на **дискретные речевые юниты** вместо спектрограмм: S2UT-модель (Lee et al., 2021) значительно сокращает длину целевой последовательности и делает обучение и инференс эффективнее за счёт использования HuBERT-юнитов вместо непрерывного сигнала.
- **UnitY** ввёл двухпроходную схему: между декодерами добавлен T2U-энкодер, который переводит текст в юниты, преодолевая разрыв между задачами.
- Современные модели уходят от авторегрессии: нерегрессивные и диффузионные архитектуры (TranSpeech, DiffS2UT, DiffuseST, DDM) повышают скорость декодирования и естественность речи.
- Новейшее направление (2025-2026) — **speech-aware LLM** поверх готовых мультимодальных моделей: SLM-S2ST построена на открытой модели Phi4-MM и при увеличении данных и размера модели достигает уровня текущих SOTA-систем.
- Для дубляжа отдельно развивается **textless-дискретно-диффузионный** подход: Dub-S2ST использует не-авторегрессивный дискретно-диффузионный декодер для юнитов вместе с их дедупликацией по скорости произношения.
- Итоговый обзорный вывод: прямые S2ST-модели сохраняют идентичность говорящего, снижают задержку и улучшают естественность за счёт сохранения просодии и голосовых характеристик, но всё ещё ограничены нехваткой данных и высокой вычислительной стоимостью.

# Оптимальная архитектура: гибридная Two-Pass Non-Autoregressive S2ST

Синтез лучших практик из UnitY / SeamlessM4T-v2 / Dub-S2ST / SLM-S2ST:

```
Исходное аудио (16kHz)
        │
┌───────▼────────────────────────────┐
│ 1. Speech Encoder                  │  Conformer / w2v-BERT 2.0 (pretrained SSL)
│    + Length Adaptor                │  ~600M–1B параметров
└───────┬─────────────────────────────┘
        │ речевые эмбеддинги (сжатые ~2×)
        │
┌───────▼─────────────────────────────┐
│ 2. Text Decoder (1st pass, auxiliary)│  NLLB-style Transformer decoder
│    → предсказывает целевой текст     │  ~1.3B параметров (используется как
│    (используется как soft-target и   │   ко-обучающий сигнал + для fallback S2TT)
│    для CTC-выравнивания)             │
└───────┬───────────────────────────────┘
        │ скрытые текстовые представления
        │
┌───────▼───────────────────────────────┐
│ 3. T2U (Text-to-Unit) Encoder-Decoder │  Non-autoregressive, CTC-based
│    предсказывает discrete speech units │  duration predictor (как в Dub-S2ST/UnitY2)
│    (HuBERT k-means, ~5000-10000 юнитов)│  ~300-500M параметров
└───────┬─────────────────────────────────┘
        │ дискретные юниты (+ speaker/prosody embedding)
        │
┌───────▼─────────────────────────────────┐
│ 4. Unit-based Vocoder (HiFi-GAN variant) │  ~15-30M параметров
│    → синтез waveform, сохранение голоса  │  conditioned on speaker embedding
└───────┬─────────────────────────────────┘
        │
   Переведённое аудио
```

**Почему именно так:**

| Компонент | Обоснование выбора |
|---|---|
| Pretrained SSL-энкодер (не с нуля) | Экономит недели self-supervised предобучения на немаркированных данных |
| Non-autoregressive T2U с CTC | Быстрый инференс (нужен для realtime/streaming), убирает экспоненциальный рост ошибок AR-декодирования |
| Text decoder как auxiliary (не единственный путь) | Даёт устойчивый градиент на старте обучения (data scarcity problem), но не создаёт задержки, т.к. на инференсе можно использовать напрямую T2U |
| Discrete units вместо спектрограмм | Кросс-энтропия вместо MSE/regression — стабильнее обучение, короче последовательности |
| Отдельный vocoder | Маленький, обучается отдельно и дёшево, не тянет весь пайплайн |
| Speaker/prosody embedding | Сохраняет голос и интонацию говорящего (ключевое преимущество direct S2ST по данным обзора выше) |

**Итоговый размер модели:** ≈ 2.2–2.5 млрд параметров (без vocoder).

# Требования к обучающим данным

По аналогии с SeamlessAlign/SpeechMatrix — нужен корпус около **400 000–500 000 часов** параллельных речевых пар (mined + synthetic TTS-augmented), покрывающих 50-100 языков.

# Аппаратные требования

| Параметр | Значение |
|---|---|
| GPU | NVIDIA H200 (141 ГБ HBM3e, ~4.8 ТБ/с bandwidth) |
| Минимум для обучения (с нуля, батч ~1B токенов/шаг) | 8× H200 (что и указано у вас) |
| VRAM на модель + оптимизатор (Adam, fp32 states) | ~2.5B параметров × ~16 байт (веса+градиенты+optimizer states в mixed precision) ≈ 40 ГБ на реплику модели |
| Активации (batch, seq_len ~ 1500 юнитов) | 150-300 ГБ суммарно при активационном чекпоинтинге |
| Сеть между узлами (если >1 сервера) | NVLink/NVSwitch внутри узла достаточно для 8 GPU; между узлами — InfiniBand 400+ Гбит/с |
| Хранилище датасета (500K часов аудио + юниты) | ~150-250 ТБ (сырое аудио) + ~5-10 ТБ (закодированные дискретные юниты) |

Ваша конфигурация (8×H200, 1.05 ТБ суммарной VRAM) — это ровно тот минимум, который используют для обучения моделей такого класса (SeamlessM4T-v2, аналогичные research-модели) без необходимости шардинга по нескольким узлам.

# Оценка скорости обучения

Это приблизительная инженерная оценка (не результат поиска — экстраполяция по известным формулам компьютерной стоимости обучения трансформеров), поэтому воспринимайте цифры как порядок величины, а не точный прогноз:

**Формула:** `FLOPs ≈ 6 × N × D`, где N — параметры модели, D — количество обучающих токенов (дискретных юнитов).

- N ≈ 2.5×10⁹ (без учёта vocoder)
- 500 000 часов аудио × ~25 юнитов/сек (после дедупликации) × 2 (источник+цель) ≈ **~90 млрд токенов** (D)
- Итого: FLOPs ≈ 6 × 2.5×10⁹ × 9×10¹⁰ ≈ **1.35 × 10²¹ FLOPs**

**Производительность кластера:**
- H200 пиковая BF16-производительность: ~989 TFLOPS/GPU
- Реалистичная утилизация (MFU) для encoder-decoder с cross-attention: ~35-40%
- Эффективная скорость: ~989 × 0.38 ≈ **375 TFLOPS/GPU** → 8 GPU ≈ **3 × 10¹⁵ FLOP/сек**

**Расчёт времени:**

```
1.35×10²¹ / 3×10¹⁵ ≈ 450 000 сек ≈ 125 часов ≈ ~5.2 суток
```

## Итоговая оценка по этапам (реалистичный сценарий):

| Этап | Что происходит | Время на 8×H200 |
|---|---|---|
| 0. Инициализация | Загрузка pretrained SSL-энкодера (w2v-BERT/HuBERT) + NLLB-декодера — **не обучается с нуля** | — |
| 1. Multitask fine-tuning (S2TT + T2U одновременно) | Основной этап на 400-500K часов данных | **~5-7 суток** |
| 2. Vocoder training | Малая модель (15-30М параметров), можно на 1-2 GPU из кластера | **~1-2 суток** (параллельно с этапом 1 на других GPU) |
| 3. Fine-tuning/alignment, RLHF-подобная калибровка просодии | Дообучение на meнее объёмных, более чистых данных | **~1-2 суток** |
| **Итого (реалистичный wall-clock)** | | **≈ 7-10 суток** |

⚠️ Если бы пришлось **также обучать speech-энкодер с нуля** (self-supervised на ~1M+ часов немаркированного аудио, как это делает Meta для w2v-BERT 2.0) — это отдельный и гораздо более дорогой этап, добавляющий **ещё 3-4 недели** на этом же кластере. Именно поэтому в индустрии почти никто не обучает энкодер заново — берут готовый чекпоинт и дообучают.
