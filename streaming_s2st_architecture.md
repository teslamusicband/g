# Streaming Speech-to-Speech Translation для звонков в реальном времени
## Переработка архитектуры с учётом фидбека: нативный стриминг + data efficiency (2k–20k часов)

## Почему предыдущая (direct) архитектура не подходит

| Критерий | Direct S2ST (предыдущий вариант) | Требование задачи (звонок) |
|---|---|---|
| Тип энкодера | w2v-BERT/HuBERT — **bidirectional**, видит весь чанк целиком | Нужен **каузальный** энкодер, обрабатывающий аудио по мере поступления |
| Стриминг | Эмулируется нарезкой на окна — задержка и разрывы контекста ("костыль") | Нативный incremental/streaming, без искусственных окон |
| Требуемые данные | ~400-500K часов параллельных **аудио-аудио** пар | Реалистично: 2K сейчас, 20K — цель прототипа |
| Единица обучения | Вся система обучается совместно (S2TT+T2U) с нуля на паре языков | Нужно переиспользовать готовые предобученные ASR/MT/TTS |
| Тип отказа при нехватке данных | Деградация всей системы целиком (единая модель) | Возможность независимо улучшать/менять модули |

Вывод: direct-архитектура объективно лучше подходит для **офлайн-дубляжа/локализации контента** (где не важна задержка и есть много данных), но плохо ложится на **живой звонок с ограниченными данными**.

## Новая архитектура: Cascaded Streaming S2ST

Каскад из трёх **независимо потоковых** модулей, каждый — дообученный (fine-tuned) поверх мощного предобученного backbone, а не обучаемый с нуля:

```
Входящий аудиопоток (чанки по 200-400ms)
        │
┌───────▼─────────────────────────────────┐
│ 1. Streaming ASR                        │  Causal Conformer / wav2vec-S backbone
│    (block-wise self-attention,          │  (предобучен на многоязычных данных,
│     KV-cache reuse между чанками,       │   дообучен на 2-20K часов русского)
│     CTC-streaming decoding)             │  → инкрементальный текст-транскрипт
└───────┬───────────────────────────────────┘
        │ частичный транскрипт (растущий по словам)
        │
┌───────▼───────────────────────────────────┐
│ 2. Streaming MT (Simultaneous Translation) │  LLM/Transformer-переводчик + read/write
│    read/write policy (alignment-based,     │  policy (StreamSpeech-style) ИЛИ
│    KV-cache reuse, syntax-aware chunking)  │  LLM+KV-cache (InfiniSST-style)
└───────┬─────────────────────────────────────┘
        │ инкрементальный перевод (по словам/фразам)
        │
┌───────▼─────────────────────────────────────┐
│ 3. Streaming TTS                            │  VoXtream/S5-TTS-style: incremental
│    (word-by-word, monotonic alignment,      │  phoneme→duration→acoustic tokens,
│     block-wise генерация, TTFB ~100ms)       │  speaker embedding для сохранения голоса
└───────┬─────────────────────────────────────┘
        │
   Исходящий переведённый аудиопоток
```

### Ключевые архитектурные решения по каждому блоку

**1. Streaming ASR**
- Backbone: каузальный Conformer или **wav2vec-S** — блочное self-attention вместо bidirectional, абсолютные синусоидальные позиционные эмбеддинги вместо relative-conv, переиспользование KV-cache между чанками (устраняет полную перекомпоновку на каждый инкремент, в отличие от ванильного wav2vec 2.0).
- Предобучение: self-supervised на большом мультиязычном корпусе (аналог XLS-R/MMS/USM, доступны открытые чекпоинты).
- Дообучение: 2-20K часов русской размеченной речи (звонки/разговорная речь) — согласно данным по низкоресурсным ASR, этого объёма достаточно при хорошем pretrained backbone.
- Выход: инкрементальный CTC-транскрипт, обновляющийся по мере поступления чанков.

**2. Streaming MT**
- Два конкурирующих варианта на выбор (рекомендуется A/B на прототипе):
  - **(a) Alignment-based read/write policy** (StreamSpeech-style / AliBaStr-MT): классический Transformer-переводчик с политикой "читать/писать", определяемой по alignment-статистике, а не по фиксированному числу слов (wait-k), что даёт прирост качества на низкой задержке.
  - **(b) LLM + KV-cache reuse** (InfiniSST-style): потоковый speech/text-энкодер подаёт чанки в LLM-переводчик, который переиспользует KV-cache предыдущих чанков — не пересчитывает контекст заново на каждом шаге.
- Критичное преимущество для data efficiency: этот блок работает **на тексте**, а не на аудио — можно взять готовую мультиязычную MT-модель (NLLB-класса или LLM) и почти не дообучать её на аудио-данных вообще, только на текстовых парах (которых для RU↔EN/др. языков на порядки больше, чем аудио-аудио пар).

**3. Streaming TTS**
- Backbone: VoXtream-style (инкрементальный phoneme-transformer → temporal-transformer для семантики/длительности → depth-transformer для акустических токенов) или S5-TTS (word-by-word с monotonic alignment learning).
- TTFB (time-to-first-byte до первого звука): ~100ms на GPU по опубликованным данным VoXtream.
- Адаптация голоса: согласно Align2Speak — дообучение под целевой голос/язык требует **единиц-десятков часов**, а не сотен тысяч.

## Почему это решает обе проблемы фидбека

| Проблема | Как решена |
|---|---|
| "BERT-like не стримится" | Все три блока построены на **каузальных** (block-wise/unidirectional) архитектурах с KV-cache reuse — это нативный стриминг, а не нарезка на окна поверх bidirectional-модели. |
| "200к часов не найти, есть 2к, цель 20к" | Каждый блок инициализируется из мощного **предобученного** backbone (self-supervised ASR, мультиязычная MT, мультиязычный TTS) и **дообучается**, а не обучается с нуля. Исследования (пп. 8-12 в таблице выше) показывают устойчивые результаты при дообучении на 4-300 часах для отдельных задач — 2-20K часов на весь каскад более чем достаточно. |
| Скейлинг на новые языки/домены | Модули независимы: можно улучшать/переобучать ASR, MT и TTS **по отдельности**, не трогая остальной каскад — то, что явно недоступно в direct-архитектуре (единая модель). |

## Компромисс, который нужно принять осознанно

По данным нескольких работ (Franceschini et al. via arXiv:2308.03415, arXiv:2508.13358): **каскадные системы обычно выигрывают по качеству (BLEU/точность), но проигрывают по задержке** end-to-end/direct системам, так как состоят из нескольких последовательных вычислений. Смягчается за счёт:
- Инкрементальной обработки на всех трёх этапах одновременно (конвейерная, а не последовательная задержка — пока TTS озвучивает предыдущий кусок, ASR+MT уже обрабатывают следующий);
- Легковесных backbone для real-time factor (RTF) << 1 на каждом этапе;
- Возможности впоследствии слить ASR+MT в единый speech-aware LLM (см. InfiniSST) для сокращения числа "пересадок", когда данных станет больше 20K часов.

## Рекомендация по прототипу (20K часов)

1. Взять готовые открытые предобученные backbone для всех трёх блоков (не обучать SSL с нуля — см. вывод из предыдущего анализа).
2. Дообучить Streaming ASR на 2K часах (то, что уже есть) — оценить WER на реальных звонках.
3. Streaming MT дообучать преимущественно на **текстовых** параллельных корпусах (их объём не ограничен нехваткой аудио).
4. Streaming TTS дообучать на голосовых сэмплах целевых дикторов — здесь хватит на порядок меньше данных, чем 2K часов.
5. Замерить end-to-end задержку (ASR+MT+TTS) и итеративно балансировать chunk size / lookahead в каждом блоке под целевой latency-бюджет звонка.

Ноутбук готов и протестирован end-to-end (все формы тензоров сходятся, обучение и streaming-инференс отрабатывают без ошибок на уменьшенной конфигурации). Структура:

1. **13 статей с arXiv** с пояснением, что взято в новую (каскадную) архитектуру
2. Streaming ASR: каузальный block-wise self-attention энкодер + KV-cache + CTC
3. Streaming MT: read/write policy (alignment-based) + KV-cache reuse на target-стороне
4. Streaming TTS: phoneme → duration → TemporalTransformer (causal, KV-cache) → DepthTransformer (RQ-style, авторегрессия по кодбукам нейрокодека внутри кадра)
5. `CascadeStreamingS2ST` — оркестратор, гоняющий все три модуля по чанкам с общим streaming-состоянием
6. Synthetic-датасеты + независимые training loops на каждый модуль
7. Демонстрация streaming-инференса чанк-за-чанком с логами
8. Замер latency/RTF по каждому модулю
9. Итоговая сводка: что заменить перед реальным деплоем (реальные backbone wav2vec-S/NLLB/VoXtream, реальный нейрокодек, production KV-cache) и план на 2K→20K часов
